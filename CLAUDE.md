# OL Linx Agent — Execution Repo

Knowledge store + execution loops for Omri's PM work at Linx Security.
**Strategy and sparring live elsewhere:** the claude.ai Linx project (`brain/claude-project-instructions.md`) and the installed role skills (positioning-linx, prd-partner-linx, discovery-interview-linx — managed via `ops/omri-linx-skills.md`). This repo is the hands: capture, loops, knowledge.

## Session start

1. The SessionStart hook (`.claude/hooks/session-start-check.sh`) reports orphan branches and log staleness. Treat its warnings as authoritative; surface them before substantive work.
2. Read `commitments.md`. Mention overdue items once — don't nag.
3. `brief.md` is generated by `loops/morning-brief` when active. If its date is stale, say so and don't rely on it.
4. `log.md` is read-on-demand only: tail the last 40 lines (`offset` parameter). Never full-read — it grows unboundedly.

## Context loading — propose and proceed

`context-index.md` maps every knowledge file by tag. For any substantive task: state which files you're loading and why, name what you're skipping, then **proceed** — don't wait for confirmation unless the choice is genuinely ambiguous. `[ref]`-tier files load only when the task needs depth. Skip the proposal entirely for quick factual answers.

## Capture — the rule that prevents data loss

If it isn't committed to this repo, it doesn't exist (documented Week-1 loss: 7 orphan branches). So:

- Meeting, decision, customer signal, commitment → write to `log.md` / `commitments.md` / `customer-intel.md` **in the same turn**, not at session end.
- The Stop hook auto-commits and pushes whatever is left. It's the net, not the mechanism.
- On a feature branch, merge to `main` before ending — orphan branches are blindness next session.
- `loops/daily-extract` automates transcript capture when active.

## Data recording discipline

- Canonical entity names, one spelling per entity (Saviynt, ConductorOne, Niv Goldenberg) — grep must match deterministically.
- Atomic bullets in `knowledge/` — one fact per line, source cited.
- Dates inline (YYYY-MM-DD) on time-sensitive claims.
- Commitments: `action — date — owner`. Rationale goes to log.md, referenced as `[→ log YYYY-MM-DD]`.
- `customer-intel.md` is for **customers only** (Sony, Monday.com, …). Internal colleagues (Mor, Dor, Niv, Sarit, Amir) go to log.md and `references/stakeholder-map.md` — never customer-intel. (Caught by replay eval 2026-06-11: both old and new systems made this mistake.)

## Loops

`loops/README.md` is the registry — one success metric per loop, hours-saved/week tracked there. All schedulers ship **inactive**; activation is Omri's manual step. Every loop output opens with its generated-at timestamp so silent death is visible.

## Style

Open with the answer. Evidence or silence — "critical/significant/major" banned unless paired with a number, quote, or named consequence. No hedges. Headers and bullets only when they earn their keep. Re-read before sending; cut what can be cut.

## Code changes

Before presenting: what changed / what could break / assumptions made (one line each). Self-review for missing error handling, hardcoded values, unhandled edge cases — fix, don't flag.

## Live infrastructure — do not touch

`telegram-bot.sh` (launchd service, routes to multiple projects beyond this repo), `notify.sh`, `.claude/settings.json`. Never read, print, or commit `.env`.
